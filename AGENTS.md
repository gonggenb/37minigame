如果任务涉及玩法、代码实现、Unity 场景、Prefab、UI 或系统调整，还需要继续阅读：

1. `docs/project_core.md`
2. `docs/gameplay_systems.md`
3. `docs/unity_tech.md`
4. `docs/codex_workflow.md`

如果任务是在要求推进下一步开发、拆分任务、修复问题或生成提示词，还需要额外阅读：

5. `docs/task_prompts.md`

## 2. 项目一句话目标

本项目是一款使用 Unity 制作的“小体量、可反复游玩的 1 分钟武侠 Roguelite 自动战斗游戏”。

玩家在 60 秒主地图阶段中探索、碰怪、进入自动战斗、获得成长、寻找隐藏洞穴，并在时间结束后进入最终 Boss 战。

当前首要目标不是做完整商业游戏，而是先跑通一个可反复试玩的小型核心闭环。

## 3. 三条硬规则

以下规则不得擅自修改：

1. 普通地图碰怪战斗时，主地图 60 秒倒计时继续流逝。
2. 隐藏洞穴内战斗时，主地图 60 秒倒计时暂停。
3. 最终 Boss 战不受 60 秒限制，作为独立阶段单独计算 Boss 战时间。

## 4. Unity 制作约束

本项目使用 Unity 制作，默认使用 C# 编写游戏逻辑。

Codex 不得擅自：

- 更换游戏引擎。
- 改成 Godot、Unreal、Cocos 或其他引擎方案。
- 引入与 Unity 项目无关的工程结构。
- 假装已经完成 Unity Editor 中的手动绑定。
- 接入真实广告 SDK、支付 SDK、后端服务、账号系统、联机系统或排行榜系统。

如果某些内容需要在 Unity Editor 中手动完成，必须明确说明：

- 需要创建哪些 GameObject。
- 需要挂载哪些脚本。
- Inspector 需要绑定哪些字段。
- 需要拖入哪些 Prefab、UI、Audio 或配置引用。
- 如何进入 Play Mode 验证。

## 5. 当前开发优先级

优先级从高到低：

1. 跑通核心玩法闭环。
2. 保证 Unity 项目可运行、可编译、可测试、可反复试玩。
3. 再增加武学、敌人、洞穴、奖励、事件等内容。
4. 最后再做 UI 美化、音效、正式美术、商业化预留。

## 6. 当前不要做

除非用户明确要求，否则不要做：

- 真实广告 SDK。
- 真实支付 SDK。
- 后端服务器。
- 账号系统。
- 联机系统。
- 排行榜系统。
- 云存档。
- 大型剧情系统。
- 大型装备词条系统。
- 大规模正式美术接入。
- 大范围重构。
- 更换 Unity 技术路线。
- 更换主要编程语言。

## 7. 完成任务后的回复要求

Codex 完成任务后，需要明确回复：

- 修改了哪些文件。
- 新增了哪些文件。
- 实现了什么功能。
- 哪些内容需要在 Unity Editor 手动绑定。
- 是否符合三条核心时间规则。
- 如何运行。
- 如何测试。
- 还有哪些未完成事项。

不要只回复“已完成”。

## 8. 核心执行原则

如果任务目标和本文件冲突，以本文件为准。

如果用户的新指令明确覆盖旧规则，以用户最新明确指令为准。

如果不确定，不要擅自扩大范围，先基于当前 Unity 仓库结构做最小、安全、可运行的实现。
